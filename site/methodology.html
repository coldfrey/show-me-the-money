Methodology

Every figure on this site can be traced to a public data point and a calculation described on this page. If you find an error, we want to know.

01Where the data comes from

Everything is computed from Elexon's Insights (BMRS) open data API — the official settlement data for Great Britain's Balancing Mechanism. No API key, no scraping, no private sources. The primary dataset is the settlement stack:

GET data.elexon.co.uk/bmrs/api/v1/balancing/settlement/stack/all/bid/{date}/{period} GET data.elexon.co.uk/bmrs/api/v1/balancing/settlement/stack/all/offer/{date}/{period}

One row per accepted balancing action per settlement period (48 half-hour periods per day; 46 or 50 on clock-change days — we always request periods 1–50 and treat "not found" as empty). Station identities and fuel types come from Elexon's BMU reference list (/reference/bmunits/all). Every raw API response is archived unmodified, so every number is reproducible offline from the archive.

02What counts as a constraint action

Elexon marks each action with an soFlag: "potentially taken for reasons other than energy balancing" — in practice, transmission constraints. We count an action as a constraint action if and only if it is SO-flagged. This is the same filter wastedwind.energy uses. It is an imperfect indicator, and we treat it as such (see §07).

03Curtailment — wind paid to stop

Over a day's bid stack (offers to reduce output), we take rows where the unit's fuel type is wind and the action is SO-flagged:

wind_bids = bids where fuelType == "WIND" and soFlag cost (£) = Σ originalPrice × volume # negative price × negative volume ⇒ positive £ volume (MWh)= |Σ volume|

Wind farms usually charge to turn down (negative bid price) because curtailment costs them subsidy revenue — so the cost is usually positive. Some units bid at positive prices, meaning they refund the system when turned down. Both are included as-is. On days when refunds outweigh payments, daily curtailment cost is legitimately negative.

We use originalPrice × volume — not the transmission-loss-adjusted or post-settlement figures — because that is precisely what wastedwind.energy computes, and Phase A of this project is a faithful replication for validation. Adjusted variants may be added later as clearly-labelled alternates.

04Replacement — gas paid to start

Curtailed energy still has to be generated somewhere. For each settlement period in which wind was curtailed, we walk the offer stack (offers to increase output) and price the replacement volume:

  1. Candidates: every offer in the period not flagged as short-duration (cadlFlag), all fuel types, SO-flagged or not.
  2. Sort: non-SO-flagged offers first, then by ascending sequenceNumber (stable sort — ties keep their published order).
  3. Consume offers in that order until the consumed volume equals the period's curtailed volume. The final offer is pro-rated so volumes match exactly. If the stack runs out, we price only what exists — no extrapolation.
  4. Replacement cost for the period = Σ originalPrice × consumed volume.

The daily replacement cost is the sum over periods. This is a replacement-cost estimate: "what was actually paid, in offer-stack order, for a volume equal to the curtailed volume." It is not the same thing as "all payments to SO-flagged generators" — we compute that separately (§06) and never mix the two.

05The headline

total constraint cost = curtailment cost + replacement cost

That sum is the "paid twice" figure: consumers fund both sides of the same megawatt-hour through balancing charges on their bills. Monthly and yearly figures are plain sums of daily figures. A month still in progress is marked partial.

06Who got paid — attribution

League tables attribute payments to stations and parent companies. Payments per station are summed from SO-flagged actions (gross payment: Σ originalPrice × volume). Stations map to parent companies via a published seed table of lead-party name matches (SSE, RWE, Uniper, EDF, Ørsted, joint ventures like Seagreen = SSE / TotalEnergies, and so on). Anything unmatched keeps its lead-party name and is marked unverified — we do not guess ownership.

Tables currently badged preview use an interim aggregate: rankings from the raw settlement stack, scaled to the day's verified totals. The final per-station attribution (including signed refund rows) replaces them shortly; daily and monthly headline figures are already final-methodology.

We report gross payment, not profit. A gas plant paid £80/MWh has fuel costs; the premium-above-market framing is arguably fairer to generators but less transparent to compute, so gross is the headline and alternates belong here in the methodology.

07Validation — proving the numbers

Phase A of this project replicates wastedwind.energy's published methodology — decompiled from the site's own JavaScript bundle — line by line. We then compare our monthly aggregates against their published summary figures for complete months, requiring agreement within ±2% on each of: curtailment cost, curtailment volume, replacement cost, replacement volume. Divergences beyond that must be investigated and either fixed or explained in a public waiver. We also sanity-check against Elexon's independent settlement-cashflow endpoints.

08Known limitations — read this bit

  • Our number is a floor. BM settlement data only. Constraint actions taken as bilateral trades or through the Local Constraint Market, and curtailment of embedded (non-BM) wind, are invisible here. True constraint costs are higher.
  • The SO-flag is imperfect. It can catch non-constraint actions and miss constraint actions. We document it; we don't "correct" it.
  • Recent data is provisional. Elexon revises settlement data for days after the fact. The trailing week is re-fetched on every run; figures for the last day or two can move.
  • Data quirks are kept, not cleaned. Live settlement data contains oddities — synthetic rows with null identifiers, occasional duplicate reference entries, units with no fuel type (shown as UNKNOWN). Each is handled by a documented rule in the engine's METHODOLOGY file rather than silently dropped.
  • Batteries and negative prices are part of the story. Negative-priced offers and positive-priced bids are included as-is, exactly as settled.

09Reproduce it yourself

The entire pipeline — API client, calculations, validation harness, and this site — is a single open codebase. Every raw API response is cached verbatim; every calculation is a pure function of that cache; every day's JSON output is published. Pull the same public data and you will get the same numbers.
